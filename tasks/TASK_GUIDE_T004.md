## Requirement (Pillar 1 — Adapt the requirement)

Add a simple REST endpoint `/api/run-test` so non-MCP callers (e.g. scripts, future Harness pipeline steps) can trigger a test session without a full MCP client. The `/health` endpoint from T001 already exists — confirm it still works alongside the new route.

**Restated intent**:
> A caller can `POST /api/run-test` with `{ platform, flow_description, app_url_or_package, timeout }` and get back `{ status, report_url, screenshots, logs }`, without needing to speak MCP. Internally this reuses the same session/tool primitives built in T002/T003 — it does not duplicate logic.

**Out of scope**:
- Actual agentic NL flow interpretation server-side — this REST endpoint still requires the caller (or an orchestrating script) to know the specific navigate/click/fill steps, OR this endpoint is a thin single-navigate-and-screenshot smoke check if no NL interpreter exists server-side (per the Option A architecture decision: no server-side LLM). Confirm the exact contract with the Supervisor before implementing if ambiguous — do not guess a hidden NL-parsing behavior.
- GitHub Actions / Harness pipeline example — explicitly deferred, not required for this task per user's Phase 0 answer ("run to check on local only")

**Requirement Refs** (from `PRD.md`):
- FR-007: `/health` endpoint (already exists from T001 — verify, don't rebuild)
- Original `requirements.txt` Phase 3 intent: simple REST wrapper for non-MCP callers (kept in scope at reduced ambition — CI/pipeline integration itself is out of scope per Phase 0 Section A)

## Approach

Thin Express (or equivalent minimal Node HTTP framework already used for `/health` in T001) route that internally calls the same session-start → single-navigate-and-screenshot → session-end sequence built in T002/T003. Keep this endpoint deliberately simple — it is a convenience wrapper, not a new execution engine (Karpathy: Simplicity First, avoid duplicating T002/T003 logic).
