Decision

Problem: How should ui_run_flow interpret and execute a natural-language web test flow inside the Dockerized MCP server, and how should a Docker image/report scheme support it?
Option A — The Simple Path: primitive Playwright MCP tools + explicit session bracketing, no server-side LLM.
Meets every FR/NFR from the PRD with the least code and smallest risk surface: no server-side LLM credentials needed, deterministic and debuggable, produces clean session-scoped reports, and directly reuses the proven playwright-mcp tool shape the original requirements doc pointed at. It also satisfies the 50% Rule by avoiding an internal planning/retry loop and using the official Playwright Docker image instead of hand-rolled browser install scripting.

Assumptions & Open Questions

  • Resolved MCP transport is HTTP/SSE on localhost:8765, not a stdio bridge — confirmed by user during grilling.
  • Resolved ui_run_flow's original single-call shape is replaced by primitive tools (ui_navigate, ui_click, ui_fill, ui_assert, ui_get_page_state) plus ui_start_session/ui_end_session — confirmed by user.
  • Resolved Docker image is based on the official mcr.microsoft.com/playwright image, not manual browser install — confirmed by user.
  • Resolved Browser scope is Chromium-only for v1 — confirmed by user during grilling.
  • Resolved Failure handling is fail-fast with partial report, not best-effort continue — confirmed by user during grilling.
  • Assumption Claude Code (the calling agent) is capable of reasoning over structured page-state output well enough to drive multi-step flows without server-side planning assistance.
  • Deferred Mobile/native support (Appium + Maestro) — explicitly out of scope until v2, per Phase 0 Section B.
  • Deferred Whether the MCP SDK's built-in HTTP/SSE transport fully supports this session model — flagged for a quick spike at the start of Task 1 (T1) in Stage 2.
